# MLXBits Image Studio

A native **SwiftUI macOS app** that drives **FLUX image generation** by spawning the
`mflux` Python CLI as a subprocess. The app owns the queue, UI, persistence, and gallery;
all model inference happens in the external `mflux-generate-flux2` process.

## Swift baseline (MATCH THIS — do not emit older idioms)

- **Deployment target: macOS 26.5.** Modern APIs are available — prefer current SwiftUI; do
  NOT target macOS-14-era APIs or add availability shims for them.
- **MainActor-by-default isolation** (`SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`,
  `SWIFT_APPROACHABLE_CONCURRENCY = YES`). Types are main-actor isolated unless you say otherwise;
  do not sprinkle redundant `@MainActor` where it's already the default, and mark genuinely
  off-actor work explicitly (`nonisolated`, `@concurrent`, or an explicit `Task.detached`).
- **State: the Observation framework** — `@Observable` classes, never `ObservableObject` /
  `@Published`. Stores are `final class`es.
- Views read state via `@Environment(SomeStore.self)`, never `@EnvironmentObject`.
- Concurrency is `async/await` + structured `Task`.

## Project layout

The Xcode project is **generated by XcodeGen from `project.yml`** — never hand-edit the
`.xcodeproj`. Source is added by directory, so new files are picked up automatically.

```
App/        @main entry (MLXBitsImageStudioApp) + ContentView (root split layout)
Models/     FluxJob (the request), FluxModelCatalog, LoraEntry, PromptTemplate
Runner/     FluxJobRunner — spawns mflux, streams stdout, parses progress
Stores/     AppSettings, JobStore (queue+persistence), GalleryStore (scans output dir)
Views/      ParamsPanel/ (inputs), PreviewPane/ (live+completed image), Gallery/,
            Queue/, Settings/, Shared/ (reusable controls)
Utilities/  BinaryDetector, JobProgressParser, KeychainHelper, MetadataSidecar,
            MfluxInstaller, UvInstaller
Resources/  Info.plist, entitlements
```

## Core architecture

- **App root** (`MLXBitsImageStudioApp`) constructs four `@State` stores and injects them
  into the environment: `AppSettings`, `JobStore`, `GalleryStore`, `FluxJobRunner`.
- **JobStore** is the authoritative queue of `FluxJob`s. It serializes to
  `~/Library/Application Support/MLXBits Image Studio/jobs.json` after every mutation.
  It does NOT execute jobs.
- **FluxJobRunner** executes jobs one at a time: picks the next pending `FluxJob`, spawns the
  `mflux-generate-flux2` `Process`, streams output into the job's `log`, updates status, then
  calls `runNext(in:settings:)` to drive the queue forward. Cancellation = `process.terminate()`.
- **AppSettings** persists preferences; every property `save()`s in its `didSet`. The
  **HuggingFace token lives in the Keychain** (`KeychainHelper`), never in JSON.
- **GalleryStore** is the source of truth for generated-image history (scans the output dir +
  `MetadataSidecar` JSON next to each image).

## Generation flow

1. User configures a `FluxJob` in `ParamsPanelView` → `JobStore.add` (or `addBatch`).
2. `FluxJobRunner.runNext` spawns `mflux` with CLI args derived from the job + resolved
   `ModelDefaults`.
3. `JobProgressParser` turns stdout lines into progress; stepwise previews stream to a cache dir.
4. On completion the image lands in the output dir with a metadata sidecar; `GalleryStore` picks it up.

## Gotchas / house rules

- **Codable snapshots: new fields MUST be `Optional` with a `?? default`.** A non-optional new
  field makes the whole decode throw, silently wiping the user's saved jobs/settings on next
  launch. Use `decodeIfPresent` (see `LoraEntry`, `JobStatus`).
- `JobStatus` is a manually-`Codable` enum (string tag + optional message) — extend it the same way.
- Long-running work stays off the main thread, but **store mutations happen on the main actor**
  (which is the default isolation here).
- Signing: Release builds use `DEVELOPMENT_TEAM = 39TQC8LANW` (paid, for distribution). Debug
  stays unsigned. `release.sh` archives Release.

## Workflow expectations

- After editing Swift, run `swiftlint lint --quiet` and fix new violations, **then** `xcodebuild`
  before considering the change done. `.swiftlint.yml` is the source of truth for style.
- This app is a sibling of **MLXBits Video Studio** (LTX-2.3 video gen); they share the
  `@Observable` store + subprocess-CLI architecture and the same macOS 26.5 / MainActor-default
  Swift baseline. Keep patterns consistent across the two where it makes sense.
- One concern per change; keep diffs focused.
